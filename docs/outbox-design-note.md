# Outbox 설계 메모

## 왜 서비스마다 Outbox 엔티티를 따로 두나 (중복 유지)

- 공통 라이브러리로 빼는 순간 서비스 간 결합이 생긴다.
- 부작용: 라이브러리 스키마/필드 변경이 여러 서비스 동시 수정/재배포 이슈로 번질 수 있다.
- 판단 기준: MSA에서는 코드 중복보다 서비스 독립성 훼손 리스크가 더 크다.

## 그래서 어떻게 운영하나

- 공통 필드(`eventId`, `aggregateType`, `aggregateId`, `eventType`, `payload`, `status`, `partitionKey`)는 문서/컨벤션으로 통일한다.
- 구현 코드는 각 도메인(Order/User/Auth..) 별 로 둔다.
- 필드 확장이 필요하면 해당 도메인에서만 변경하고, 다른 도메인은 영향 없이 독립 배포한다.

## 팀 내 합의 문장

- 코드 중복은 감수 가능하다.
- 서비스 간 결합은 더 위험하다.
- 공통 라이브러리로 빼고싶은 충동이 생겨도 참아주세요
